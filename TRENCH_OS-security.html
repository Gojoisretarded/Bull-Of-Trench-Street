The client renders. The server decides.

This game has a token attached, so an economy bug isn’t cheating — it’s theft. Every balance, price, and PnL number the browser can touch, the browser can forge. This is the threat model to build against, ranked by blast radius, with the fix for each.

Critical — economy / theft High — abuse / integrity Medium — hardening
00 / THE ONE RULE THAT KILLS MOST BUGS

If the browser can assert it, the browser can fake it

Balances, prices, PnL, clout, mining yield — all computed and stored server-side. The client is only ever allowed to request an action, never report an outcome.

GROUND TRUTH

The original stored the wallet balance in localStorage.

That’s editable in three seconds from the dev console — infinite money, for free. It’s the single biggest thing to not repeat. Your wallet lives in Postgres, guarded by the tick service; the tab holds nothing but a cached view it cannot forge.

01 / THE THREAT MODEL

What to defend against, worst first

Nothing here exists to exploit yet — it’s the checklist of attacks the architecture must make impossible by design.

■ Critical · economy & theft
01

Client-authoritative money critical

Balance / price / PnL held or asserted by the client → edit it, mint infinite currency. The original’s fatal flaw.

Money state lives only server-side. Client sends intents (“buy 2 of $CMR”), server validates funds and computes the result.

02

The real-money bridge crown jewel

The instant in-game currency converts to real SOL, every economy bug becomes a withdrawal exploit.

Treat cash-out as a hardened boundary: signature verification, withdrawal limits, time delays, manual review thresholds. Keep funny-money and on-chain value separate until you deliberately bridge them.

03

Double-spend / race conditions critical

Fire 10 concurrent sells before the first settles → spend currency you don’t have.

DB transactions + row-level locks on the wallet, idempotency keys on every trade, atomic balance checks. No read-then-write gaps.

04

Trade input validation critical

Negative amounts, overflow, or executing against a stale/spoofed price.

Schema-validate every intent (Zod/TypeBox). Server re-reads the live price at execution — never trusts a client-supplied price or amount.

■ High · abuse & integrity
05

Fake “verified” PnL high

If the flex/larp system trusts a client-submitted number or screenshot, everyone verifies as real and the mechanic dies.

Real PnL is computed server-side from the ledger — the client cannot claim it. Larp = an explicitly unverified post.

06

Market manipulation high

Wash trading (self-buys faking volume), Sybil accounts pumping one coin, front-running the tick.

Per-account + per-IP limits, wash-trade detection, launch cooldowns, and never expose the next tick to clients.

07

Stored XSS in user content high

Coin names, usernames, and feed posts render to everyone. A coin named <script> steals sessions.

Escape/sanitize all user content on output, strict Content-Security-Policy, and never dangerouslySetInnerHTML on user data.

08

WebSocket trust high

Unauthenticated sockets, clients subscribing to channels they shouldn’t, spoofed WS messages.

Authenticate on connect, authorize every channel subscription, treat all inbound socket messages as untrusted intents.

09

AI / NPC proxy abuse high

Prompt injection, jailbreaks, and cost abuse — spamming NPC chats to run up your Claude bill.

Key stays server-side only. Per-user rate + spend caps, capped history/token length, input moderation, claude-haiku-4-5 for cheap chatter.

■ Medium · hardening
10

Auth & recovery codes medium

The frictionless recovery-code save is an account-takeover vector if codes are guessable.

High-entropy codes, hashed at rest, rate-limited redemption, no account enumeration.

11

SQL injection medium

String-built queries let an attacker read or wipe the ledger.

Parameterized queries / an ORM (Prisma or Drizzle). Never concatenate SQL.

12

CSRF medium

Cookie sessions with credentials: include (what the original uses) are CSRF-able.

SameSite cookies + CSRF tokens, tight CORS allowlist.

13

Mining / time exploits medium

“Claim every 20 min” checked client-side = infinite money via clock manipulation.

Accrual computed from server timestamps only; claims are idempotent.

14

Secrets leakage medium

API keys or DB creds bundled into the client build.

Nothing sensitive in the frontend. Env vars server-side, and scan the bundle before every ship.

02 / BAKE IT INTO THE BUILD

Each phase ships with its own defenses

Security isn’t a phase at the end — it’s a checklist attached to every phase of the roadmap. Don’t advance until the box is ticked.

1

The first trade

  • Server-authoritative wallet — balance never leaves Postgres
  • Schema validation + idempotency keys on the trade endpoint
  • Row locks / transactions so concurrent trades can’t double-spend
2

Create + living market

  • Price re-read server-side at execution; next tick never exposed
  • Mining accrual from server time only
  • Sanitize coin names/tickers (XSS) at creation
3

Feed + flex/larp

  • PnL verified from the ledger, not client-submitted
  • Output-escape all posts + strict CSP
  • Redis rate limits on posting; wash-trade / Sybil detection
4

Characters + NPCs

  • Claude key server-side only; per-user spend + rate caps
  • Cap conversation history/tokens; moderate NPC inputs
  • Authenticated + authorized WebSocket channels
5

Polish + launch

  • Recovery codes: high-entropy, hashed, rate-limited
  • CSRF + CORS lockdown; secrets scan on the bundle
  • If cashing out to real SOL: harden the withdrawal bridge — limits, delays, signature checks, review
